== Learning outcomes

After this workshop you can:

* Explain why _before-after averages mislead_
* Draw a DAG for a proposed engineering intervention
* Estimate an ATE and a CACE from cluster-randomised data
* Compute a required sample size and a minimum detectable effect
* Apply an O'Brien-Fleming boundary when you inspect an experiment repeatedly

// ============================================================================

== The problem

Your organisation rolled out an AI reviewer. Review latency dropped from 8.4 hours to 5.9 hours. The vendor renewal is next week.

*Did the tool cause the improvement?*

Try to answer with what you already have. Notice how easy it is to fall for the naive answer.

// ============================================================================

== The five alternative explanations

. Quieter release period during the pilot
. Repositories with simpler work adopted first
. Reviewer staffing changed
. CI became faster at the same time
. Novelty effect from a shiny new tool

Naive comparison _can't rule any of them out_.

// ============================================================================

== Potential outcomes

For unit latexmath:[$i$]:

* latexmath:[$Y_i(1)$] = outcome _with_ treatment
* latexmath:[$Y_i(0)$] = outcome _without_ treatment
* Individual effect latexmath:[$\tau_i = Y_i(1) - Y_i(0)$]

*The fundamental problem:* you only observe one.

Every causal method is a strategy for _estimating the missing counterfactual_.

== Assignment vs analysis unit

* *Assignment unit* — receives the treatment (team, repository, engineer)
* *Analysis unit* — contributes an outcome (pull request, Engineering Change, service)

They can differ. Interference must be considered.

Pull-request assignment breaks down when reviewers are shared. Team-level assignment costs statistical power. Engineering-Change assignment is often the best match to the mechanism.

// ============================================================================

== Difference-in-differences


For treated (T) and control (C) groups before/after:

[stem]
++++
\widehat{\tau}_{DiD} = (\bar{Y}_{T,\text{after}} - \bar{Y}_{T,\text{before}}) - (\bar{Y}_{C,\text{after}} - \bar{Y}_{C,\text{before}})
++++

*Key assumption:* parallel trends. Absent treatment, the two groups would have moved similarly.

Inspect pre-treatment trends. Report sensitivity.

// ============================================================================

== The AI reviewer worked example

* latexmath:[$\bar{Y}_{T,\text{before}} = 5.6h$], latexmath:[$\bar{Y}_{T,\text{after}} = 4.9h$]
* latexmath:[$\bar{Y}_{C,\text{before}} = 6.4h$], latexmath:[$\bar{Y}_{C,\text{after}} = 6.8h$]
* DiD: latexmath:[$(4.9 - 5.6) - (6.8 - 6.4) = -1.1h$]

Not the −1.9h the naive read suggested.

// ============================================================================

== Complier-average causal effect

Only 71% of treated teams actually opened AI findings.

*Intention-to-treat:* effect of the _policy_ that assigns the tool.

*CACE:* effect _among those who complied_.

[stem]
++++
\widehat{\tau}_{CACE} = \frac{\widehat{\tau}_{ITT}}{P(\text{compliance})} = \frac{-1.1}{0.71} = -1.55h
++++

*Which one goes on the exec slide?* Depends on the decision — mandate vs. voluntary rollout.

== Statistical power

Power = probability of detecting an effect _when it exists_.

For a two-sample z-test on means:

[stem]
++++
n = 2 \left( \frac{(z_{\alpha/2} + z_\beta) \sigma}{\delta} \right)^2
++++

Required n grows with variance. Shrinks with expected effect.

_Rare severe defects_ + _cluster-correlated units_ + _heavy tails_ = larger n than teams expect.

// ============================================================================

== Minimum detectable effect

A negative result is worthless without an MDE.

* "No effect detected at 500 changes per arm" ≈ "no effect > 15% detectable"
* Same result at 32,000 per arm ≈ "no effect > 2% detectable"

Power analyses belong in the _pre-registration_, not the discussion.

== Sequential testing

Engineering experiments run continuously. Teams _peek_.

Repeated naive tests at α = 0.05 inflate false positives fast:

* 1 look: 5%
* 5 looks: ~14%
* 10 looks: ~19%

*Solution:* group-sequential boundaries that spend α over planned looks.

// ============================================================================

== O'Brien-Fleming boundary

At K planned looks:

[stem]
++++
z_k = z_{\alpha/(2K)} \sqrt{K/k}
++++

Very strict early. Approaches conventional α by the final look.

* Look 1 of 5: reject if latexmath:[$|z| > 4.56$]
* Look 3 of 5: reject if latexmath:[$|z| > 2.63$]
* Look 5 of 5: reject if latexmath:[$|z| > 2.04$]

// ============================================================================

// ============================================================================

== Don'ts, worth memorising

* Don't rely on before-after alone. Bring a control group or a plausible counterfactual.
* Don't launch without a pre-analysis plan and pre-declared MDE.
* Don't peek without a boundary.
* Don't confuse ITT and CACE in the executive summary.
* Don't retire a Class 2+ metric without a versioned replacement.

// ============================================================================

== Wrap-up

* Draw a DAG before you launch
* Pick the estimand that matches the decision
* Compute MDE _before_ collecting data
* Peek only under a boundary
